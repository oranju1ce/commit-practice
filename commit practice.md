Git команды:

1. Настройка и конфигурация

    1. 1. git config

    Сотни вещей в Git работают без всякой конфигурации, используя параметры по умолчанию. Для большинства из них вы можете задать иные умолчания, либо вовсе использовать собственные значения. Это включает в себя целый ряд настроек, начиная от вашего имени и заканчивая цветами в терминале и вашим любимым редактором. Команда config хранит и читает настройки в нескольких файлах, так что вы можете задавать значения глобально или для конкретных репозиториев.
    Команда git config используется практически в каждой главе этой книги.

    В разделе Первоначальная настройка Git главы 1 мы использовали эту команду для указания имени, адреса электронной почты и редактора ещё до того, как начать использовать Git.

    В разделе Псевдонимы в Git главы 2 мы показали, как можно использовать её для создания сокращённых вариантов команд с длинными списками опций, чтобы не печатать их все каждый раз.


    1. 2. git help

    Команда git help служит для отображения встроенной документации Git о других командах. И хотя мы приводим описания самых популярных команд в этой главе, полный список параметров и флагов каждой команды доступен через git help <command>.

2. Клонирование и создание репозиториев

    Существует два способа создать Git репозиторий. Первый — клонировать его из существующего репозитория (например, по сети); второй — создать репозиторий в существующем каталоге.

    2. 1. git init

    Чтобы превратить обычный каталог в Git репозиторий и начать версионировать файлы в нём, просто запустите git init.

    Это «точка старта». git init превращает обычную папку на твоём компьютере в локальный Git-репозиторий.
    Что происходит внутри?
    Когда ты вводишь эту команду, Git создает в папке скрытую директорию .git. Именно там хранится вся история изменений, настройки и служебная информация. Если ты удалишь эту папку, проект перестанет быть репозиторием, и вся история коммитов исчезнет (хотя сами файлы в папке останутся).


    2. 2. git clone

    На самом деле git clone работает как обёртка над некоторыми другими командами. Она создаёт новый каталог, переходит внутрь и выполняет git init для создания пустого репозитория, затем она добавляет новый удалённый репозиторий (git remote add) для указанного URL (по умолчанию он получит имя origin), выполняет git fetch для этого репозитория и, наконец, извлекает последний коммит в ваш рабочий каталог, используя git checkout.

    git clone нужна, чтобы скопировать уже существующий репозиторий (например, с GitHub, GitLab или Bitbucket) к себе на компьютер.В отличие от git init, которая создает пустой репозиторий с нуля, clone скачивает всё: файлы, все ветки и всю историю изменений (коммитов).

    Полезные приёмы:
    Клонирование в конкретную папку:
    Если не хочешь, чтобы папка называлась как проект, укажи имя в конце:
    git clone https://link.com my-folder-name

    Клонирование по SSH (стандарт для работы):
    Чтобы не вводить логин и пароль каждый раз, DevOps используют SSH-ключи:
    git clone git@github.com:user/project.git

    Клонирование без истории (Shallow Clone):
    Если проект огромный, а тебе нужно только собрать Docker-образ или запустить тесты (не нужна вся история за 10 лет), используй флаг --depth:
    git clone --depth 1 https://link.com

3. Основные команды

    Всего несколько команд нужно для базового варианта использования Git для ведения истории изменений.

    3. 1. git add

    Команда git add добавляет содержимое рабочего каталога в индекс (staging area) для последующего коммита. По умолчанию git commit использует лишь этот индекс, так что вы можете использовать git add для сборки слепка вашего следующего коммита.
    это мост между твоим рабочим каталогом и будущим коммитом. Она переводит файлы из состояния Modified или Untracked в состояние Staged (индекс).
    Представь, что ты собираешь посылку: ты можешь положить в коробку только часть вещей со стола. git add — это как раз процесс укладывания конкретных вещей в коробку перед тем, как её запечатать (сделать commit).


    3. 2. git status

    Команда git status показывает состояния файлов в рабочем каталоге и индексе: какие файлы изменены, но не добавлены в индекс; какие ожидают коммита в индексе. Вдобавок к этому выводятся подсказки о том, как изменить состояние файлов.

    Что именно она сообщает:
    На какой ветке ты находишься (например, On branch main).

    Синхронизация с удаленным репозиторием: опережаешь ли ты GitHub на несколько коммитов или, наоборот, отстаешь от него.

    Changes to be committed (Изменения для коммита): файлы, которые ты уже добавил через git add. Они подсвечиваются зеленым.

    Changes not staged for commit (Изменения, не подготовленные для коммита): файлы, которые ты изменил, но еще не добавил в индекс. Они подсвечиваются красным.
    
    Untracked files (Неотслеживаемые файлы): совершенно новые файлы, о которых Git еще ничего не знает. Тоже красные.


    3. 3. git diff
    
    Команда git diff используется для вычисления разницы между любыми двумя Git деревьями. Это может быть разница между вашей рабочей копией и индексом (собственно git diff), разница между индексом и последним коммитом (git diff --staged), или между любыми двумя коммитами (git diff master branchB).
    это команда-рентген. Она показывает конкретные строки кода, которые ты добавил, изменил или удалил. Если git status говорит «какой» файл изменился, то git diff показывает — «как» именно.

    Вот основные сценарии её использования:

        1. Просмотр незакоммиченных изменений (Working Directory)

        Если ты просто изменил файл и хочешь увидеть разницу до того, как добавишь его в индекс:
    
        git diff

        Что покажет:
        разницу между твоим текущим кодом на диске и тем, что уже находится в индексе (Staging Area).
        Цвета:
        добавленные строки обычно зеленые (со знаком +), удаленные — красные (со знаком -).
    

        2. Просмотр подготовленных изменений (Staging Area)

        Если ты уже сделал git add, обычный git diff ничего не покажет. Чтобы увидеть, что именно сейчас «лежит в коробке» и готово к коммиту:

        git diff --staged

        (Или аналогичная команда git diff --cached).
   

        3. Сравнение веток

        В DevOps это критично при проверке кода (Code Review) перед слиянием. Посмотреть разницу между твоей веткой и основной:
   
        git diff main my-feature-branch


        4. Сравнение между коммитами
        Если хочешь увидеть, что изменилось между двумя точками в истории:

        git diff <hash_коммита_1> <hash_коммита_2>


    3. 4. git difftool

    Команда git difftool просто запускает внешнюю утилиту сравнения для показа различий в двух деревьях, на случай если вы хотите использовать что-либо отличное от встроенного просмотрщика git diff.
    это продвинутая версия git diff. Вместо того чтобы выводить изменения в виде текстового полотна прямо в терминале, она запускает внешнюю графическую программу (GUI), которая наглядно подсвечивает разницу.


    3. 5. git commit

    Команда git commit берёт все данные, добавленные в индекс с помощью git add, и сохраняет их слепок во внутренней базе данных, а затем сдвигает указатель текущей ветки на этот слепок.
    это команда, которая делает «снимок» (snapshot) состояния твоего проекта. Если git add — это упаковка вещей в коробку, то git commit — это заклеивание коробки скотчем и наклеивание на неё бирки с описанием.

    Как правильно использовать:
        Базовый вариант:
        git commit -m "Добавил скрипт для бэкапа базы данных"
        Флаг -m (message) позволяет написать комментарий прямо в терминале. Без него Git откроет текстовый редактор по умолчанию (например, Vim), чтобы ты вписал сообщение там.
        
        Добавить изменения и сразу закоммитить (Fast Track):
        Если ты только менял существующие файлы (не создавал новых) и хочешь пропустить шаг git add, используй флаг -a:
        
        git commit -am "Поправил порт в конфигурации Nginx"

        Исправить последний коммит (Amend):
        Это очень полезная фишка. Допустим, ты сделал коммит, но забыл добавить один файл или опечатался в комментарии. Вместо создания нового «мусорного» коммита, сделай:
        
        git commit --amend -m "Новое правильное сообщение"
        
        Важно: используй это, только если ты еще не отправил (push) коммит на GitHub.

    
    3. 6. git reset

    Команда git reset, как можно догадаться из названия, используется в основном для отмены изменений. Она изменяет указатель HEAD и, опционально, состояние индекса. Также эта команда может изменить файлы в рабочем каталоге при использовании параметра --hard, что может привести к потере наработок при неправильном использовании, так что убедитесь в серьёзности своих намерений прежде чем использовать его.
    это мощный инструмент для «отката» изменений. Она позволяет переместить указатель ветки (HEAD) на любой предыдущий коммит, фактически переписывая локальную историю.

        Три режима работы (самое важное)
        Выбор режима зависит от того, что ты хочешь сделать с изменениями, которые были накоплены после целевого коммита.
        Режим   | Куда денутся изменения?                                  | Безопасность
        --soft  | Останутся в индексе (готовы к новому коммиту)            | ✅ Безопасно
        --mixed | (по умолчанию) Останутся в файлах, но выпадут из индекса | ✅ Безопасно
        --hard  | Удалятся навсегда (и из индекса, и из файлов)            | ⚠️ Опасно


    3. 7. git rm

    Команда git rm используется в Git для удаления файлов из индекса и рабочей копии. Она похожа на git add с тем лишь исключением, что она удаляет, а не добавляет файлы для следующего коммита.
    Команда git rm используется для удаления файлов из-под контроля Git. Просто удалить файл через «Корзину» или командой rm в терминале недостаточно — Git заметит пропажу и пометит файл как «deleted» в статусе, но его всё равно нужно будет закоммитить. git rm делает это за один шаг.


    3. 8. git mv

    Команда git mv — это всего лишь удобный способ переместить файл, а затем выполнить git add для нового файла и git rm для старого.

    Команда git mv используется для перемещения или переименования файлов и папок в твоем репозитории.
    Хотя ты можешь просто перетащить файл мышкой в папку или переименовать его через правую кнопку, использование git mv делает это «официально» для Git, сокращая количество твоих действий.

    Что она делает на самом деле?
        Когда ты запускаешь git mv file_old.txt file_new.txt, Git выполняет за тебя три шага:
        1. Переименовывает файл в твоей рабочей папке.
        2. Удаляет старое имя из индекса (git rm --cached).
        3. Добавляет новое имя в индекс (git add).


    3. 9. git clean

    Команда git clean используется для удаления мусора из рабочего каталога. Это могут быть результаты сборки проекта или файлы конфликтов слияний.

    Команда git clean — это «пылесос» для твоего репозитория. Она удаляет из рабочей директории все неотслеживаемые (untracked) файлы.
    В работе DevOps-инженера это незаменимая вещь, когда нужно быстро очистить проект от «мусора»: временных логов, скомпилированных файлов, артефактов сборки или кэшей, которые ты создал случайно и не хочешь видеть в git status.

    ⚠️ Важное предупреждение
    Эта команда удаляет файлы физически. В отличие от git reset, здесь нет «безопасного режима» — если ты удалишь что-то лишнее, Git не сможет это восстановить, так как эти файлы никогда не были в коммитах.

    Основные флаги (как не удалить лишнего)
    Git очень осторожен, поэтому просто команда git clean обычно выдает ошибку. Нужно обязательно указать флаг:
        1. Тестовый прогон (-n или --dry-run):
        Самая важная команда. Она просто покажет список того, что будет удалено, но ничего не тронет.
        2. Удаление файлов (-f или --force):
        Принудительное удаление всех неотслеживаемых файлов.
        3. Удаление папок (-d):
        По умолчанию clean не трогает пустые или полные папки. Чтобы вычистить и их.
        4. Удаление игнорируемых файлов (-x):
        Обычно clean не трогает файлы, прописанные в .gitignore. Но если ты хочешь сбросить проект до идеально чистого состояния (например, удалить все установленные библиотеки в node_modules или логи), добавь -x.

        Интерактивный режим (-i)
        Если ты хочешь вручную выбрать, что удалить, а что оставить.
        Git предложит меню, где можно помечать файлы на удаление или фильтровать их.

4. Ветвление и слияния

    За создание новых веток и слияние их воедино отвечает несколько Git команд.

    4. 1. git branch
    
    Команда git branch — это своего рода "менеджер веток". Она умеет перечислять ваши ветки, создавать новые, удалять и переименовывать их.
    Команда git branch — это один из главных инструментов DevOps-инженера. В Git ветки (branches) представляют собой просто легкие подвижные указатели на конкретные коммиты. Они позволяют изолировать разработку разных фич, исправление багов или настройку CI/CD пайплайнов друг от друга.
    
    Основные сценарии использования

        1. Просмотр существующих веток

            Посмотреть список локальных веток:

            git branch

            Текущая ветка, на которой ты находишься, будет подсвечена цветом и отмечена звездочкой *.


            Посмотреть локальные и удаленные (на GitHub) ветки разом:

            git branch -a


        2. Создание новой ветки
    
            Чтобы создать ветку (например, с именем feature-nginx), выполни:

            git branch feature-nginx

            Важно: эта команда только создает ветку, но не переключает тебя на неё автоматически.
    

        3. Переименование ветки
        
            Переименовать текущую ветку, на которой ты стоишь:

            git branch -m new-branch-name


        4. Удаление веток (уборка мусора)
    
            Когда ветка уже слита с основной и больше не нужна, её стоит удалить, чтобы не захламлять репозиторий:

            Безопасное удаление (Git выдаст ошибку, если в ветке есть неперенесенные изменения):

            git branch -d feature-nginx


            Принудительное удаление (стереть в любом случае):

            git branch -D feature-nginx

    
    Полезные приемы и DevOps-контекст
    
        Создание и переход за один шаг:

            Обычно ветку создают и сразу переключаются на неё.
            Старый способ: git checkout -b <имя>. 
            Современный и рекомендуемый способ:

            git switch -c feature-nginx


        Связь с удаленным репозиторием:
        
            Если ты создал ветку локально и хочешь отправить её на GitHub, обычный git push не сработает. Нужно связать её с сервером с помощью флага -u (--set-upstream):

            git push -u origin feature-nginx

    
    4. 2. git checkout

    Команда git checkout используется для переключения веток и выгрузки их содержимого в рабочий каталог.
    Команда git checkout — это классический швейцарский нож в Git. Исторически она выполняла сразу две огромные задачи: переключение между ветками и восстановление (откат) файлов.

    Два главных сценария использования:

        1. Работа с ветками и коммитами (Перемещение)
        
            Переключение на другую ветку:

            git checkout feature-nginx

            Git обновляет файлы в твоей рабочей папке, чтобы они соответствовали состоянию ветки feature-nginx.


            Создание новой ветки и переход на неё за один шаг:

            git checkout -b feature-docker

            (Флаг -b означает branch).

        
            «Временное путешествие» к конкретному коммиту:

            git checkout <хэш_коммита>

            Ты перейдешь в состояние «detached HEAD» (отсоединенный HEAD). Ты сможешь посмотреть, каким был код три дня назад, запустить его и протестировать. Изменения здесь делать нельзя (они потеряются), но для дебага в DevOps это незаменимо. Чтобы вернуться назад, просто введи git checkout main.

        2. Работа с файлами (Откат изменений)

            Если ты изменил файл, еще не добавил его в индекс (git add), но понял, что написал полную ерунду, и хочешь вернуть всё как было в последнем коммите:

            git checkout -- config.yaml

            Два дефиса -- нужны, чтобы Git точно понял: дальше идет имя файла, а не название ветки.

        
        Применение в DevOps и CI/CD
        
            В пайплайнах (например, в Jenkins, GitLab CI или GitHub Actions) шаг получения кода часто выглядит именно так:

            git clone <url>
            git checkout <нужная_ветка_или_тег_версии>


            Это гарантирует, что сборка Docker-образа или деплой на прод-сервер будут происходить строго из правильного среза кода.


    4. 3. git merge

    Команда git merge используется для слияния одной или нескольких веток в текущую. Затем она устанавливает указатель текущей ветки на результирующий коммит.
    Команда git merge используется для слияния двух или более веток. Она берет историю изменений из одной ветки (например, feature-nginx) и интегрирует ее в ту ветку, на которой ты сейчас находишься (например, main).
    
    В DevOps-инженерии слияние веток — это ключевой этап автоматизации. Именно в момент слияния кода в основную ветку обычно триггерятся CI/CD пайплайны, которые запускают тесты и деплоят код на сервера.

    Как выполнить слияние (Базовый алгоритм)

        Если ты хочешь влить изменения из ветки feature в ветку main:

            1. Переключись на ту ветку, в которую хочешь влить код:

            git switch main

            2. Запусти слияние, указав ветку, откуда берешь код:

            git merge feature


    Два сценария слияния

        1. Fast-Forward (Перемотка вперед)

            Этот сценарий происходит автоматически, если с момента создания ветки feature в ветке main не появилось новых коммитов.

                Как это работает:
                Git просто перемещает указатель ветки main вперед, на самый последний коммит ветки feature.

                Результат:
                История остается идеально прямой, никаких дополнительных коммитов слияния не создается.
